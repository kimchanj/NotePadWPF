# WinMemo (Windows 메모장 클론) 개발 스펙 v0.1.0

> 문서 목적: 포트폴리오 공용 템플릿을 기반으로 **Windows 메모장(Notepad)** 수준의 텍스트 편집기를 구현하기 위한 **개발 스펙/PRD/설계 문서**를 한 파일로 제공한다.

## 0. 프로젝트 개요 (Overview)

- **프로젝트명(상품형 이름)**: WinMemo (윈메모) — Windows 메모장 클론
- **한 줄 설명(고객가치)**: 가볍고 빠른 **.txt/.md** 편집 + 자동저장 + 탭/최근파일을 제공하는 데스크톱 메모장
- **타겟 사용자(고객군)**:
  - 개발자/학생/사무직: 간단한 기록, 로그 확인, 마크다운 메모
  - “메모장은 너무 단순, IDE는 과함”인 사용자
- **문제 요약(고객 Pain Point)**:
  - 윈도우 기본 메모장은 기능이 제한적(탭/자동저장/최근파일 관리 등 부족)
  - 대용량 텍스트에서 느려지거나 작업이 끊김(환경에 따라)
  - .md 편집 시 최소한의 편의(미리보기/문법 강조)가 없음
- **핵심 해결 가치(3줄)**:
  1) **빠른 실행/편집 성능** + UI 멈춤 최소화  
  2) **.md 친화 기능**(문법 강조/미리보기 옵션)  
  3) **실사용 기능**(탭, 최근파일, 자동저장/복구, 검색/치환)

### 0-1. 범위/버전

- **MVP 범위(필수)**:
  - 단일 창 + 탭 기반 문서 편집(.txt/.md)
  - 열기/저장/다른 이름으로 저장
  - 최근 파일 목록
  - 찾기/다음찾기/바꾸기
  - 줄 번호 토글(옵션)
  - 인코딩 선택(UTF-8/UTF-8 BOM/ANSI) + 줄바꿈(CRLF/LF)
  - 변경 감지(Dirty) + 닫기/종료 시 확인
  - 자동저장/비정상 종료 복구(초기 버전은 단순 캐시 방식)
- **V2 확장 범위(옵션)**:
  - 마크다운 미리보기(분할 뷰)
  - 문법 강조 강화(코드블록, 링크, 체크박스 등)
  - 다중 창, 드래그&드롭, 파일 탐색기 컨텍스트 메뉴 등록
  - 플러그인/테마(다크/라이트), 커스텀 폰트 프리셋
- **제외 범위(이번 버전에서 안함)**:
  - 협업(동시편집), 클라우드 동기화
  - 노트 앱 수준의 데이터베이스(태그/검색 DB) 중심 구조
  - IDE급 기능(디버깅, 프로젝트 탐색 등)

### 0-2. 입력/출력 정의

- **입력(데이터/파일/폼/API)**:
  - 로컬 파일: `.txt`, `.md`, `.log`, `.json`(텍스트로만), 기타 텍스트 확장자
  - 사용자 입력: 편집기 텍스트, 설정(폰트/인코딩/줄바꿈/자동저장)
- **출력(파일/화면/리포트/API)**:
  - 저장 파일: `.txt/.md` 등
  - (옵션) 자동저장 캐시 파일: `*.autosave`
  - UI 상태: 탭, 변경 표시, 검색 결과 하이라이트
- **저장/이력(옵션)**:
  - 설정 파일: `settings.json`
  - 최근 파일 리스트: `recent.json`
  - 복구 세션: `session.json` + 임시 텍스트 캐시

### 0-3. 기술스택(도구함)

> 목표: “윈도우 네이티브 느낌 + 배포 쉬움 + UI 생산성” 조합

- **UI/클라이언트**: C# .NET 8 + WPF (또는 WinUI 3 대안)
- **백엔드/서비스(있다면)**: 없음(로컬 앱)
- **데이터 처리/분석**: 없음(텍스트 처리 유틸)
- **DB/저장소**: 로컬 JSON + 파일 시스템
- **문서/리포트**: README.md, 릴리즈 노트
- **배포**: MSIX 또는 단일 exe(자체 포함 배포), GitHub Releases
- **기타(로깅/모니터링/테스트)**:
  - 로깅: Serilog (파일 로그)
  - 테스트: xUnit (핵심 유틸/파서/설정 로더)

## 1. 요구사항 작성 (Requirements)

### 1-1. 사용자 요구사항 (User Stories)

- US01. 사용자는 텍스트 파일을 열고 빠르게 편집하고 저장할 수 있어야 한다.
- US02. 사용자는 여러 파일을 탭으로 열어 전환하며 작업할 수 있어야 한다.
- US03. 사용자는 실수/크래시로 내용이 날아가지 않도록 자동저장/복구가 되어야 한다.
- US04. 사용자는 찾기/바꾸기로 긴 문서에서 원하는 텍스트를 빠르게 찾을 수 있어야 한다.
- US05. 사용자는 .md를 편집할 때 최소한의 문법 강조(또는 미리보기)를 받고 싶다(옵션).

### 1-2. 기능 요구사항 (Functional Requirements)

- FR01. 파일 열기/저장/다른 이름으로 저장 지원
- FR02. 탭 기반 다중 문서 편집(새 탭/닫기/전체 닫기)
- FR03. 변경사항(Dirty) 표시 및 닫기/종료 시 저장 확인
- FR04. 찾기/다음 찾기/이전 찾기/바꾸기/모두 바꾸기
- FR05. 인코딩(UTF-8, UTF-8 BOM, ANSI) 선택 저장
- FR06. 줄바꿈(CRLF/LF) 선택 저장 + 파일 로드 시 감지
- FR07. 최근 파일 목록(최대 N개) 및 빠른 재오픈
- FR08. 자동저장(간격 설정) + 비정상 종료 시 복구 안내
- FR09. 드래그&드롭으로 파일 열기
- FR10. (옵션) .md 문법 강조 / 분할 미리보기

### 1-3. 비기능 요구사항 (Non-Functional Requirements)

- NFR01. 성능:
  - 앱 실행 1초 내(일반 PC 기준 목표)
  - 5MB 텍스트 편집 시 기본 타이핑 지연이 체감되지 않도록(목표: 프리즈 없음)
- NFR02. 안정성:
  - 자동저장으로 크래시/강제종료에도 최대 1분 이내 손실로 제한(기본값)
- NFR03. 사용성:
  - 기본 메모장 수준의 단순한 UI + 단축키 제공(Ctrl+S, Ctrl+F 등)
  - “저장 안 됨” 상태가 항상 눈에 보이도록
- NFR04. 보안(필요 시):
  - 네트워크 통신 없음
  - 로컬 파일 권한만 사용(관리자 권한 불필요)
- NFR05. 호환성/환경:
  - Windows 10/11
  - 고DPI 스케일링 대응

### 1-4. 제약사항/가정 (Constraints & Assumptions)

- C01. 외부 서버/클라우드 없이 로컬 파일만 다룬다.
- C02. 문서 편집은 “텍스트” 중심이며 바이너리 편집은 지원하지 않는다.
- A01. 사용자는 기본 단축키/메뉴 구조에 익숙하다고 가정한다(메모장/VSCode 경험).

### 1-5. 성공 기준 (Acceptance Criteria)

- AC01. `.txt`/`.md` 파일을 열고 수정 후 저장했을 때, 내용/인코딩/줄바꿈 옵션이 정확히 반영된다.
- AC02. 탭 3개 이상 열고 전환해도 입력 지연이 체감되지 않는다(일반 문서 기준).
- AC03. 앱을 강제 종료 후 재실행하면 자동저장 캐시를 감지하고 복구 선택을 제공한다.

## 2. 기획안 (Planning / PRD)

### 2-1. 문제 정의 (Problem Statement)

- 현재 프로세스:
  - 기본 메모장 또는 IDE로 간단 메모/로그 확인
- 문제점:
  - 기본 메모장은 기능 부족(탭/자동복구/최근파일 정교함/마크다운 편의)
  - IDE는 무겁고 실행/전환 비용 큼
- 영향(시간/비용/오류):
  - 작업 흐름이 자주 끊김(파일 탐색/재오픈 반복)
  - 저장 실수로 작업 손실 위험

### 2-2. 목표 & KPI (Goals)

- 목표1: “메모장만큼 가볍게” 열고 쓰게 만들기
- 목표2: “실사용 편의” (탭/최근파일/자동복구/검색치환) 제공
- KPI(측정 가능 지표):
  - 앱 실행 시간: 1초 내 목표(릴리즈 노트에 측정 환경 명시)
  - 자동복구로 인한 데이터 손실 최대치: 기본 60초
  - 최근파일 재오픈: 2클릭 이내

### 2-3. 사용자 시나리오/플로우 (User Flow)

1) 앱 실행  
2) 최근 파일 클릭 또는 파일 열기(Ctrl+O)  
3) 편집(탭 전환 가능)  
4) 저장(Ctrl+S) 또는 자동저장  
5) 종료 시 저장 여부 확인 / 크래시 시 복구 제안

### 2-4. 화면/UX 기획 (Wireframe 수준)

- **메인 화면 구성**:
  - 상단: 메뉴바(파일/편집/보기/도움말), 탭바
  - 중단: 텍스트 편집기(스크롤, 선택, 하이라이트)
  - 하단: 상태바(줄/열, 인코딩, 줄바꿈, 변경상태, 줌)
- **상태/피드백(로딩/진행률/알림)**:
  - 큰 파일 로드 시 “로딩 스피너 + 취소(옵션)”
  - 자동저장 시 상태바에 “Autosaved HH:MM:SS”
- **에러 안내 정책**:
  - 파일 권한/잠김 → “저장 실패: 권한/다른 프로그램 사용 중” + 해결 안내
  - 인코딩 감지 실패 → “UTF-8로 열었음(변경 가능)”

### 2-5. 정책/규칙(업무룰) 정의

- Rule01: 문서가 변경되면 탭에 `●` 표시, 저장 시 제거
- Rule02: 자동저장 파일은 원본을 덮어쓰지 않고 별도 캐시에 저장
- Rule03: 복구는 “자동저장 캐시가 원본보다 최신일 때”만 제안

### 3-2. 기능 상세 명세 (각 기능별)

#### F01. 파일 열기
- 목적: 로컬 텍스트 파일을 읽어 탭으로 연다.
- 입력: `path`, (옵션) `forcedEncoding`
- 처리:
  - 파일 존재/권한 체크
  - 인코딩 감지(간단 규칙: BOM 우선, 실패 시 UTF-8 시도, 실패 시 ANSI)
  - 줄바꿈 감지(CRLF/LF)
  - 탭 생성 및 상태바 업데이트
- 출력: 탭 1개 생성, 편집기 텍스트 로드
- 예외/오류 메시지:
  - `E_OPEN_NOT_FOUND`: 파일이 없습니다.
  - `E_OPEN_DENIED`: 파일을 열 권한이 없습니다.
  - `E_OPEN_LOCKED`: 다른 프로그램에서 사용 중입니다.
- 완료 조건: 텍스트가 화면에 표시되고 커서 입력이 가능

#### F02. 저장/다른 이름으로 저장
- 목적: 편집 내용을 지정된 인코딩/줄바꿈으로 파일에 쓴다.
- 입력: `text`, `path`, `encoding`, `newlineMode`
- 처리:
  - newline 정규화(LF/CRLF)
  - encoding 적용하여 저장
  - 성공 시 Dirty 해제, 최근파일 업데이트
- 출력: 파일 저장
- 예외/오류 메시지:
  - `E_SAVE_DENIED`: 저장 권한이 없습니다.
  - `E_SAVE_IO`: 저장 중 오류가 발생했습니다.
- 완료 조건: 파일이 정상적으로 저장되고 탭 상태가 “저장됨”으로 표시

#### F03. 탭 관리
- 목적: 문서를 여러 개 열고 빠르게 전환한다.
- 입력: 새 탭, 닫기, 전체 닫기, 탭 전환
- 처리:
  - 닫기 시 Dirty면 확인(저장/저장안함/취소)
  - 세션 저장(옵션)
- 출력: 탭 상태 변화
- 예외/주의: 탭 닫기 루프에서 확인 다이얼로그 UX 깔끔하게

#### F04. 찾기/바꾸기
- 목적: 검색어를 하이라이트하고 치환한다.
- 입력: `query`, `matchCase`, `wholeWord`, `direction`, `replaceText`
- 처리:
  - 현재 커서 기준 다음/이전 검색
  - 바꾸기/모두 바꾸기(Undo 스택 고려)
- 출력: 선택 영역 이동/치환
- 예외/주의: 큰 문서에서 반복 검색 성능(인덱싱은 V2)

#### F07. 자동저장/복구
- 목적: 작업 손실 방지.
- 입력: `autosaveIntervalSec`, `sessionId`
- 처리:
  - Dirty 문서만 캐시에 저장(원본 파일과 분리)
  - 앱 시작 시 캐시 존재 + 원본보다 최신이면 복구 제안
  - 복구 후 캐시 정리(또는 유지)
- 출력: 캐시 파일, 복구 다이얼로그
- 예외/주의: 캐시 용량 관리(최대 문서 수/크기 제한)

### 3-3. 에러/메시지 규격 (Message Catalog)

- E_OPEN_NOT_FOUND: [열기] → “파일이 없습니다.” / log: `open:not_found path=...`
- E_OPEN_DENIED: [열기] → “파일을 열 권한이 없습니다.” / log: `open:denied ...`
- E_OPEN_LOCKED: [열기] → “다른 프로그램에서 사용 중입니다.” / log: `open:locked ...`
- E_SAVE_DENIED: [저장] → “저장 권한이 없습니다.” / log: `save:denied ...`
- E_SAVE_IO: [저장] → “저장 중 오류가 발생했습니다.” / log: `save:io ...`

## 4. 분석/설계 (Analysis & Design)

### 4-1. 아키텍처 개요

- 데이터 흐름(간단):
  - 파일 입력 → DocumentModel 생성 → 편집 → 저장/캐시 → 설정/최근파일 갱신

### 4-2. 폴더/프로젝트 구조

- `/WinMemo`
  - `/WinMemo.App`
    - `/UI` (Views, Dialogs, Controls)
    - `/ViewModels`
    - `/Services`
    - `/Models`
    - `/Utils`
    - `/Resources` (icons, styles)
    - `App.xaml`, `MainWindow.xaml`
  - `/WinMemo.Tests`
    - `FileServiceTests.cs`

### 4-3. 데이터 설계

- **DocumentModel**
  - `Id: Guid`
  - `FilePath: string?` (새 문서면 null)
  - `DisplayName: string`
  - `Text: string`
  - `IsDirty: bool`
  - `Encoding: string` (utf-8 / utf-8-bom / ansi)

  - Autosave interval: 10~600초 범위(기본 60)
  - Recent file max: 1~50(기본 20)

### 4-4. 파일/DB 설계

- 설정 파일 규격:
  - `%AppData%/WinMemo/settings.json`
- 최근 파일 규격:
  - `%AppData%/WinMemo/recent.json`
- 세션/복구:
  - `%LocalAppData%/WinMemo/session.json`
  - `%LocalAppData%/WinMemo/autosave/{docId}.autosave`
- 파일명 규칙:
  - autosave: `{docId}_{timestamp}.autosave` (또는 최신 1개 유지)

### 4-5. 외부 연동 설계(있다면)
- 없음(MVP)

### 4-6. 성능/동시성/백그라운드 처리

- UI 멈춤 방지 방식:
  - 파일 로드/저장은 `async` + 백그라운드 Task
  - 큰 파일은 스트리밍 로드(초기에는 전체 로드, V2에서 개선)
- 진행률 업데이트 방식:
  - 큰 파일 로드 시 읽기 바이트 기반 진행률(옵션)

### 4-7. 로깅/설정/에러 리포트

- 로그 레벨: Info(기본), Debug(개발), Error(예외)

### 4-8. 테스트 데이터(샘플) 설계

## 8. 포트폴리오 패키징 (등록/홍보용 산출물)

### 8-1. 포트폴리오 소개글(문제→해결→효과)

- 문제: 기본 메모장은 기능이 부족하고, IDE는 가볍게 쓰기엔 과하다.
- 해결: 탭/최근파일/자동저장/복구/검색치환을 갖춘 “가벼운 메모장” 구현.
- 효과(정량): 파일 재오픈/전환 시간 단축, 작업 손실 리스크 감소(오토세이브).
- 스크린샷/데모: 탭 3개 편집, 검색/치환, 자동복구 화면
- 내가 담당한 부분: 전체(설계/구현/테스트/배포)
- 유지보수/확장: MD 미리보기/테마/플러그인으로 확장 가능

### 8-2. 스크린샷 리스트(최소 5장)

1) 시작 화면(빈 문서)
2) 파일 3개 탭으로 연 화면
3) 찾기/바꾸기 다이얼로그
4) 상태바(인코딩/줄바꿈/줄·열)
5) 자동복구 제안 화면

### 8-4. 상품화 문구(크몽용)

- 상품 제목 후보 5개:
  1) “윈도우 메모장보다 편한 탭 메모장 제작”
  2) “자동저장/복구 되는 회사용 경량 편집기”
  3) “.md 편집에 최적화된 윈도우 텍스트 에디터”
  4) “로그/텍스트 빠르게 보는 업무용 메모장”
  5) “Notepad 대체 경량 편집기 커스터마이징”
- 제공 범위: MVP 기능 구현 + 설정/최근파일 + 배포 패키지
- 필요한 고객 자료: UI 선호(색/폰트), 필수 단축키 목록, 배포 방식
- 납기/수정 정책(초안): 1차 7일, 수정 2회
